Project 2 - MVVM Login

Referensi: Arsitektur MVVM

Konsep yang Digunakan

Arsitektur MVVM

MVVM (Model - View - ViewModel) membagi kode menjadi tiga lapisan:

  • Model - Data dan akses data: User (entity), UserDao, AppDatabase, UserRepository.
  • ViewModel - Logika bisnis dan state UI: LoginViewModel. Tidak mengetahui keberadaan View secara langsung, hanya mengekspos state.
  • View - UI: LoginScreen, RegisterScreen, DashboardScreen. Hanya menampilkan state dan meneruskan aksi user ke ViewModel.

Alur data satu arah: View memanggil fungsi ViewModel → ViewModel menjalankan coroutine ke Repository → Repository mengakses DAO/Room → hasil dikembalikan sebagai StateFlow → View mengamati dengan collectAsStateWithLifecycle() dan merender ulang otomatis.

Room Database

Abstraction layer di atas SQLite. Terdiri dari tiga komponen: @Entity (model tabel), @Dao (antarmuka query), dan @Database (jembatan ke file SQLite). Anotasi diproses oleh KSP (Kotlin Symbol Processing) saat kompilasi untuk menghasilkan implementasi konkret.

Kotlin Coroutines + Flow

Coroutines memungkinkan operasi asinkron (seperti query database) tanpa memblokir main thread. Fungsi suspend dapat "dijeda" sambil menunggu hasil tanpa memblokir thread. StateFlow adalah aliran data reaktif yang selalu memiliki nilai terbaru dan bisa diobservasi oleh UI.

Sealed Class untuk UI State

Sealed class merepresentasikan semua kemungkinan state secara eksplisit. Compiler Kotlin memaksa penanganan semua state dalam blok when.

Penjelasan Kode

data/local/entity/User.kt

@Entity(tableName = "users")
data class User(
    @PrimaryKey(autoGenerate = true) val id: Int = 0,
    val username: String,
    val password: String,
    val email: String,
    val createdAt: Long = System.currentTimeMillis()
)

@Entity(tableName = "users") memberitahu Room bahwa class ini merepresentasikan tabel bernama users. Setiap field menjadi kolom tabel.

@PrimaryKey(autoGenerate = true) menandai id sebagai primary key dengan auto-increment. Default 0 agar tidak perlu diisi manual — Room menggantinya dengan ID baru.

createdAt: Long = System.currentTimeMillis() menyimpan timestamp pembuatan akun. Nilai default diisi otomatis sehingga tidak perlu dipass saat membuat objek User.

data/local/dao/UserDao.kt

@Dao
interface UserDao {
    @Insert(onConflict = OnConflictStrategy.ABORT)
    suspend fun insertUser(user: User): Long

    @Query("SELECT * FROM users WHERE username = :username AND password = :password LIMIT 1")
    suspend fun login(username: String, password: String): User?

    @Query("SELECT * FROM users WHERE username = :username LIMIT 1")
    suspend fun getUserByUsername(username: String): User?

    @Query("SELECT COUNT(*) FROM users")
    suspend fun getUserCount(): Int
}

@Dao — Room menggenerate implementasi konkret dari interface ini menggunakan KSP saat kompilasi.

suspend fun — Semua fungsi DAO adalah suspend karena dipanggil dari dalam Coroutine. Ini memastikan operasi database tidak berjalan di main thread.

User? (nullable) — Jika query tidak menemukan baris yang cocok, Room mengembalikan null. Ini cara idiomatik Kotlin untuk "data tidak ditemukan".

OnConflictStrategy.ABORT — Jika ada konflik primary key, operasi dibatalkan dan melempar exception.

LIMIT 1 — Membatasi hasil query hanya 1 baris, cukup untuk keperluan login dan cek username.

data/local/AppDatabase.kt

@Database(entities = [User::class], version = 1, exportSchema = false)
abstract class AppDatabase : RoomDatabase() {
    abstract fun userDao(): UserDao

    companion object {
        @Volatile private var INSTANCE: AppDatabase? = null

        fun getInstance(context: Context): AppDatabase {
            return INSTANCE ?: synchronized(this) {
                Room.databaseBuilder(context.applicationContext, AppDatabase::class.java, "mvvm_login_db")
                    .addCallback(object : Callback() {
                        override fun onCreate(db: SupportSQLiteDatabase) {
                            super.onCreate(db)
                            CoroutineScope(Dispatchers.IO).launch {
                                INSTANCE?.userDao()?.insertUser(
                                    User(username = "demo", password = "demo123", email = "demo@example.com")
                                )
                            }
                        }
                    })
                    .build().also { INSTANCE = it }
            }
        }
    }
}

Singleton Pattern:

  • @Volatile — Memastikan nilai INSTANCE selalu dibaca dari memori utama, bukan cache CPU. Penting di lingkungan multi-thread.
  • synchronized(this) — Hanya satu thread yang bisa masuk blok ini sekaligus, mencegah dua instance dibuat bersamaan.
  • INSTANCE ?: synchronized { ... } — Pola double-checked locking: cek pertama tanpa lock (cepat), jika null baru masuk synchronized.
  • context.applicationContext — Mencegah memory leak; tidak menyimpan referensi ke Activity yang bisa dihancurkan.

addCallback.onCreate — Dijalankan sekali saat database pertama kali dibuat. Digunakan untuk memasukkan akun demo (demo/demo123) agar langsung bisa dicoba.

data/repository/UserRepository.kt

class UserRepository(private val userDao: UserDao) {

    suspend fun login(username: String, password: String): User? {
        return userDao.login(username, password)
    }

    suspend fun register(username: String, password: String, email: String): Result<Unit> {
        return try {
            val existing = userDao.getUserByUsername(username)
            if (existing != null) {
                Result.failure(Exception("Username '$username' sudah digunakan"))
            } else {
                userDao.insertUser(User(username = username, password = password, email = email))
                Result.success(Unit)
            }
        } catch (e: Exception) {
            Result.failure(e)
        }
    }
}

Repository adalah single source of truth — ViewModel tidak pernah mengakses DAO langsung. Manfaatnya: jika sumber data diganti (misal dari Room ke API), hanya Repository yang perlu diubah.

Result<Unit> adalah tipe kembalian dari Kotlin Standard Library untuk membungkus hasil yang bisa berhasil (Result.success) atau gagal (Result.failure). Lebih elegan dari melempar exception ke ViewModel.

Pengecekan username duplikat (getUserByUsername) adalah contoh logika bisnis di Repository, bukan di DAO maupun ViewModel.

viewmodel/LoginViewModel.kt

sealed class LoginUiState {
    object Idle : LoginUiState()
    object Loading : LoginUiState()
    data class Success(val user: User) : LoginUiState()
    data class Error(val message: String) : LoginUiState()
}

sealed class RegisterUiState {
    object Idle : RegisterUiState()
    object Loading : RegisterUiState()
    object Success : RegisterUiState()
    data class Error(val message: String) : RegisterUiState()
}

class LoginViewModel(private val repository: UserRepository) : ViewModel() {

    private val _loginState = MutableStateFlow<LoginUiState>(LoginUiState.Idle)
    val loginState: StateFlow<LoginUiState> = _loginState.asStateFlow()

    fun login(username: String, password: String) {
        if (username.isBlank() || password.isBlank()) {
            _loginState.value = LoginUiState.Error("Username dan password tidak boleh kosong")
            return
        }
        viewModelScope.launch {
            _loginState.value = LoginUiState.Loading
            val user = repository.login(username.trim(), password.trim())
            _loginState.value = if (user != null) LoginUiState.Success(user)
                                 else LoginUiState.Error("Username atau password salah")
        }
    }

    fun register(username: String, password: String, confirmPassword: String, email: String) {
        when {
            username.isBlank() || password.isBlank() || email.isBlank() ->
                _registerState.value = RegisterUiState.Error("Semua field wajib diisi")
            password.length < 6 ->
                _registerState.value = RegisterUiState.Error("Password minimal 6 karakter")
            password != confirmPassword ->
                _registerState.value = RegisterUiState.Error("Password tidak cocok")
            !email.contains("@") ->
                _registerState.value = RegisterUiState.Error("Format email tidak valid")
            else -> viewModelScope.launch {
                _registerState.value = RegisterUiState.Loading
                val result = repository.register(username.trim(), password.trim(), email.trim())
                _registerState.value = if (result.isSuccess) RegisterUiState.Success
                                       else RegisterUiState.Error(result.exceptionOrNull()?.message ?: "Gagal")
            }
        }
    }
}

Sealed class — Mirip enum tetapi setiap subclass bisa memiliki data berbeda. Kompiler memaksa kita menangani semua kemungkinan dalam blok when, sehingga tidak ada state yang terlewat.

Enkapsulasi state: _loginState (private MutableStateFlow) hanya bisa diubah oleh ViewModel. UI menerima loginState (StateFlow read-only via asStateFlow()).

Validasi sinkron — Pengecekan input kosong dilakukan sebelum coroutine karena tidak perlu akses database.

viewModelScope.launch — Coroutine terikat lifecycle ViewModel; otomatis dibatalkan saat ViewModel dihancurkan, mencegah memory leak.

_loginState.value = LoginUiState.Loading — UI merespons dengan menampilkan CircularProgressIndicator dan menonaktifkan tombol.

ui/screens/LoginScreen.kt

val loginState by viewModel.loginState.collectAsStateWithLifecycle()

LaunchedEffect(loginState) {
    if (loginState is LoginUiState.Success) {
        onLoginSuccess((loginState as LoginUiState.Success).user.username)
        viewModel.resetLoginState()
    }
}

Button(
    onClick = { viewModel.login(username, password) },
    enabled = loginState !is LoginUiState.Loading
) {
    if (loginState is LoginUiState.Loading) {
        CircularProgressIndicator(modifier = Modifier.size(20.dp))
    } else {
        Text("Masuk")
    }
}

collectAsStateWithLifecycle() — Mengonversi StateFlow menjadi State<T> Compose. WithLifecycle berarti pengumpulan berhenti otomatis saat UI tidak aktif (app di-background), menghemat resource.

LaunchedEffect(loginState) — Side-effect yang berjalan dalam Coroutine setiap kali loginState berubah. Navigasi adalah side-effect, tidak boleh langsung di dalam fungsi composable.

State-driven UI — Tampilan dikendalikan sepenuhnya oleh state. Tombol otomatis disabled saat Loading, menampilkan spinner atau teks tergantung state.

onLoginSuccess adalah callback lambda dari MainActivity. Ini prinsip "event hoisting" — screen tidak perlu tahu tentang navigasi secara langsung.

ui/screens/DashboardScreen.kt

@Composable
fun DashboardScreen(username: String, onLogout: () -> Unit) {
    Column(...) {
        Card(colors = CardDefaults.cardColors(containerColor = MaterialTheme.colorScheme.primaryContainer)) {
            Text("Username: $username")
            Text("Status: Aktif")
        }
        OutlinedButton(onClick = onLogout) { Text("Keluar") }
    }
}

Layar ini tidak menerima ViewModel karena hanya menampilkan data statis (username yang sudah diketahui). username dikirim sebagai parameter biasa dari navigasi.

MaterialTheme.colorScheme.primaryContainer — Menggunakan warna dari theme aktif. Ini memastikan tampilan konsisten dan otomatis mendukung dark/light mode.

MainActivity.kt

val db = AppDatabase.getInstance(this)
val repository = UserRepository(db.userDao())
val viewModel = ViewModelProvider(this, LoginViewModel.Factory(repository))[LoginViewModel::class.java]

NavHost(navController = navController, startDestination = "login") {
    composable("login") {
        LoginScreen(
            viewModel = viewModel,
            onLoginSuccess = { username ->
                navController.navigate("dashboard/$username") {
                    popUpTo("login") { inclusive = true }
                }
            },
            onNavigateToRegister = { navController.navigate("register") }
        )
    }
    composable("register") {
        RegisterScreen(viewModel = viewModel, onBack = { navController.popBackStack() }, ...)
    }
    composable("dashboard/{username}") { backStack ->
        val username = backStack.arguments?.getString("username") ?: "User"
        DashboardScreen(username = username) {
            navController.navigate("login") {
                popUpTo("dashboard/$username") { inclusive = true }
            }
        }
    }
}

Manual Dependency Injection — Rantai dependency dibangun secara eksplisit: AppDatabase → UserRepository → LoginViewModel via Factory. Ini adalah cara sederhana tanpa framework DI seperti Hilt.

class Factory : ViewModelProvider.Factory — Diperlukan karena ViewModel tidak bisa memiliki constructor dengan parameter selain SavedStateHandle secara default.

popUpTo("login") { inclusive = true } — Menghapus layar login dari back stack setelah berhasil masuk. Ini mencegah user kembali ke layar login dengan menekan tombol Back dari Dashboard.

navController.popBackStack() — Kembali ke layar sebelumnya dengan menghapus layar saat ini dari back stack.

Project 4 - Coffee Bliss (Digital Membership App)

Referensi: Mini Project Evaluasi Akhir

Proyek evaluasi akhir yang menggabungkan seluruh materi: MVVM, Room Database dengan multiple entitas dan foreign key, Jetpack Compose, dan Navigation Compose.

Konsep yang Digunakan

Room dengan Multiple Entity dan Foreign Key

Proyek ini memiliki 3 tabel yang saling berelasi: members, transaksi, dan redemptions. Relasi antar tabel dijaga dengan @ForeignKey. Dengan onDelete = CASCADE, jika member dihapus maka semua transaksi dan redemption miliknya ikut terhapus otomatis.

Reaktif dengan flatMapLatest + combine

flatMapLatest meluncurkan Flow baru setiap kali trigger berubah (misal: member login). combine menggabungkan beberapa Flow sekaligus — setiap kali salah satu berubah, hasilnya langsung dikirim ke UI. Bersama-sama, dua operator ini membuat UI selalu sinkron dengan database tanpa polling manual.

Navigation Compose dengan NavHost

Navigasi berbasis rute string dengan parameter dinamis. popUpTo membersihkan back stack agar user tidak bisa kembali ke layar login setelah masuk, atau ke layar main setelah logout.

Sistem Poin dan Status Membership

Setiap transaksi menambah poin (belanja / 10.000). Status member diperbarui otomatis berdasarkan total poin: Silver (0–199), Silver Plus (200–499), Gold (500+).

    val nama: String,
    val email: String,
    val noHp: String,
    val nomorMember: String,

totalPoin dan status memiliki nilai default. Keduanya di-update bersamaan setiap kali transaksi ditambahkan, menggunakan member.copy(totalPoin = newTotal, status = newStatus). nomorMember tidak auto-generate oleh Room — diisi oleh Repository dengan format CB0001.

data/local/entity/Transaksi.kt dan Redemption.kt

@Entity(
    tableName = "transaksi",
    foreignKeys = [ForeignKey(
        entity = Member::class,
        parentColumns = ["id"],
        childColumns = ["memberId"],
        onDelete = ForeignKey.CASCADE
    )]
)
data class Transaksi(
    @PrimaryKey(autoGenerate = true) val id: Int = 0,
    val memberId: Int,
    val jumlahBelanja: Long,
    val poinDidapat: Int,
    val createdAt: Long = System.currentTimeMillis()
)

@ForeignKey mendefinisikan relasi: kolom memberId di tabel transaksi merujuk ke kolom id di tabel members. onDelete = CASCADE berarti Room menambahkan ON DELETE CASCADE ke DDL, sehingga penghapusan member otomatis menghapus semua transaksinya di level database.

Redemption menggunakan struktur yang sama — foreign key ke members dengan CASCADE delete. Menyimpan nama reward, poin yang digunakan, dan timestamp penukaran.

data/local/dao/MemberDao.kt

@Dao
interface MemberDao {
    @Insert(onConflict = OnConflictStrategy.ABORT)
    suspend fun insertMember(member: Member): Long

    @Update
    suspend fun updateMember(member: Member)

    @Query("SELECT * FROM members WHERE email = :email LIMIT 1")
    suspend fun getMemberByEmail(email: String): Member?

    @Query("SELECT * FROM members WHERE id = :id")
    fun getMemberById(id: Int): Flow<Member?>

    @Query("SELECT COUNT(*) FROM members")
    suspend fun getMemberCount(): Int
}

getMemberByEmail adalah suspend (one-shot) — digunakan untuk login dan cek duplikasi email saat registrasi.

getMemberById mengembalikan Flow<Member?> (reaktif) — setiap kali data member berubah (poin bertambah, status naik), UI otomatis diperbarui tanpa query ulang.

getMemberCount digunakan oleh Repository untuk menghitung nomor member berikutnya.

TransaksiDao dan RedemptionDao menggunakan Flow<List<...>> agar daftar di UI selalu sinkron dengan database.

Tambah transaksi + update poin dan status:

suspend fun addTransaksiAndUpdatePoin(memberId: Int, jumlahBelanja: Long): Result<Unit> {
    val member = memberDao.getMemberByEmail(...) ?: return Result.failure(...)
    val poin = (jumlahBelanja / 10000).toInt()
    val newTotal = member.totalPoin + poin
    val newStatus = when {
        newTotal >= 500 -> "Gold"
        newTotal >= 200 -> "Silver Plus"
        else -> "Silver"
    }
    transaksiDao.insertTransaksi(Transaksi(memberId = memberId, jumlahBelanja = jumlahBelanja, poinDidapat = poin))
    memberDao.updateMember(member.copy(totalPoin = newTotal, status = newStatus))
    return Result.success(Unit)
}

Satu fungsi Repository mengeksekusi dua operasi DAO (insert transaksi + update member) secara berurutan. Logika perhitungan poin dan penentuan status ada di sini, bukan di ViewModel.

Redeem poin:

suspend fun redeemPoin(memberId: Int, namaReward: String, poinDibutuhkan: Int): Result<Unit> {
    val member = memberDao.getMemberById(memberId) ?: return Result.failure(...)
    if (member.totalPoin < poinDibutuhkan) {
        return Result.failure(Exception("Poin tidak cukup"))
    }
    redemptionDao.insertRedemption(Redemption(memberId = memberId, namaReward = namaReward, poinDigunakan = poinDibutuhkan))
    memberDao.updateMember(member.copy(totalPoin = member.totalPoin - poinDibutuhkan))
    return Result.success(Unit)
}

Validasi poin dilakukan sebelum insert. Jika lolos, poin dipotong dari total member menggunakan copy().

viewmodel/CoffeeViewModel.kt

private val _loggedInMemberId = MutableStateFlow<Int?>(null)

init {
    _loggedInMemberId
        .filterNotNull()
        .flatMapLatest { memberId ->
            combine(
                repository.getMemberById(memberId),
                repository.getTransaksi(memberId),
                repository.getRedemptions(memberId)
            ) { member, transaksi, redemptions ->
                Triple(member, transaksi, redemptions)
            }
        }
        .onEach { (member, transaksi, redemptions) ->
            _uiState.update { it.copy(
                currentMember = member,
                transaksiList = transaksi,
                redemptionList = redemptions
            )}
        }
        .launchIn(viewModelScope)
}

Cara kerja aliran reaktif ini:

  1. Awalnya _loggedInMemberId = null — tidak ada Flow yang berjalan (filterNotNull() memblokir).
  2. User login → _loggedInMemberId.value = member.id → filterNotNull() melewatkannya.
  3. flatMapLatest meluncurkan combine(3 flows) baru untuk member tersebut.
  4. combine menggabungkan data member, transaksi, redemption menjadi satu. Setiap kali salah satu berubah (poin bertambah, transaksi baru, dll), combine langsung memancarkan hasil baru.
  5. onEach mengupdate _uiState, UI langsung merender ulang.
  6. launchIn(viewModelScope) — Flow berhenti otomatis saat ViewModel di-destroy.
fun logout() {
    _loggedInMemberId.value = null
    _uiState.value = CoffeeUiState()
}

Logout cukup dengan mengisi null ke _loggedInMemberId — filterNotNull() langsung memblokir Flow, semua data reaktif berhenti, dan UI state di-reset ke kondisi awal.

MainActivity.kt

NavHost(navController = navController, startDestination = "welcome") {
    composable("welcome") {
        WelcomeScreen(
            onLogin = { navController.navigate("login") },
            onRegister = { navController.navigate("register") }
        )
    }
    composable("login") { LoginScreen(...) }
    composable("register") { RegisterScreen(...) }
    composable("main") {
        MainScreen(
            uiState = uiState,
            viewModel = viewModel,
            onLogout = {
                viewModel.logout()
                navController.navigate("welcome") {
                    popUpTo("welcome") { inclusive = true }
                }
            }
        )
    }
}

Empat rute: welcome, login, register, main. Setelah login/register berhasil, navigasi ke main dengan popUpTo("welcome") { inclusive = true } — membersihkan seluruh back stack hingga welcome (termasuk welcome). Tombol Back di layar main tidak akan membawa kembali ke login.

Saat logout, viewModel.logout() dipanggil terlebih dahulu untuk mereset state, baru navigasi kembali ke welcome dengan back stack yang bersih.

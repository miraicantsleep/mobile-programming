Project 3 - Room Database: Registrasi Siswa

Referensi: Room Database Registrasi Siswa

Konsep yang Digunakan

Room Database

Room adalah abstraction layer di atas SQLite dari Android Jetpack. Keunggulan dibanding SQLite langsung:

  • Query SQL divalidasi saat kompilasi (bukan saat runtime) — typo langsung ketahuan.
  • Mapping otomatis dari baris tabel ke objek Kotlin via anotasi @Entity dan @Dao.
  • Terintegrasi langsung dengan Kotlin Coroutines dan Flow.

Tiga komponen utama: Entity (model tabel), DAO (antarmuka query), Database (jembatan ke file SQLite).

Repository Pattern

Lapisan abstraksi antara ViewModel dan sumber data. ViewModel tidak pernah mengakses DAO langsung. Manfaat: jika sumber data berubah (Room ke API), hanya Repository yang perlu diubah. ViewModel dan UI tidak terpengaruh.

StateFlow + combine + flatMapLatest

Pola reaktif untuk menggabungkan beberapa sumber data menjadi satu UI state. flatMapLatest memastikan query lama dibatalkan saat query baru dimulai (saat user mengetik pencarian). combine menggabungkan beberapa Flow menjadi satu nilai output.

Singleton Pattern

Menjamin hanya ada satu instance database di seluruh aplikasi. Membuat instance database berulang kali sangat mahal (costly) karena membuka koneksi file SQLite.

Penjelasan Kode

data/local/entity/Siswa.kt

@Entity(tableName = "siswa")
data class Siswa(
    @PrimaryKey(autoGenerate = true) val id: Int = 0,
    val nama: String,
    val email: String,
    val createdAt: Long = System.currentTimeMillis()
)

@Entity(tableName = "siswa") — Room membuat tabel bernama siswa di SQLite. Room menghasilkan DDL berikut secara otomatis:

CREATE TABLE IF NOT EXISTS `siswa` (
    `id` INTEGER PRIMARY KEY AUTOINCREMENT NOT NULL,
    `nama` TEXT NOT NULL,
    `email` TEXT NOT NULL,
    `createdAt` INTEGER NOT NULL
)

@PrimaryKey(autoGenerate = true) — Nilai id diisi otomatis oleh SQLite (auto-increment). Default 0 agar tidak perlu diisi saat membuat objek baru.

createdAt: Long — Timestamp Unix epoch (milidetik). Digunakan untuk mengurutkan daftar dari terbaru ke terlama (ORDER BY createdAt DESC).

data class — Otomatis menghasilkan equals(), hashCode(), dan copy(). Fungsi copy() sangat penting saat update: siswa.copy(nama = "Baru") mempertahankan id dan createdAt asli.

data/local/dao/SiswaDao.kt

@Dao
interface SiswaDao {
    @Query("SELECT * FROM siswa ORDER BY createdAt DESC")
    fun getAllSiswa(): Flow<List<Siswa>>

    @Query("SELECT * FROM siswa WHERE nama LIKE '%' || :query || '%' OR email LIKE '%' || :query || '%'")
    fun searchSiswa(query: String): Flow<List<Siswa>>

    @Query("SELECT COUNT(*) FROM siswa")
    fun getSiswaCount(): Flow<Int>

    @Query("SELECT * FROM siswa WHERE id = :id")
    suspend fun getSiswaById(id: Int): Siswa?

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertSiswa(siswa: Siswa): Long

    @Update
    suspend fun updateSiswa(siswa: Siswa)

    @Delete
    suspend fun deleteSiswa(siswa: Siswa)
}

Room menggenerate kelas implementasi konkret dari interface ini saat kompilasi menggunakan KSP.

Flow<List<Siswa>> — Query reaktif: setiap kali data tabel berubah (insert/update/delete), Flow otomatis memancarkan list terbaru tanpa perlu query ulang manual. Tidak butuh suspend karena Flow sudah asinkron.

'%' || :query || '%' — Cara SQLite mengonkatenasi string. Setara dengan LIKE '%kata%'. Mencari di kolom nama dan email menggunakan OR.

@Insert(onConflict = OnConflictStrategy.REPLACE) — Jika ada primary key sama, baris lama dihapus dan diganti (upsert). Return Long adalah rowId baru.

@Update dan @Delete — Room menggenerate SQL berdasarkan primary key secara otomatis: UPDATE siswa SET ... WHERE id = ? dan DELETE FROM siswa WHERE id = ?.

data/local/AppDatabase.kt

@Database(entities = [Siswa::class], version = 2, exportSchema = false)
abstract class AppDatabase : RoomDatabase() {
    abstract fun siswaDao(): SiswaDao

    companion object {
        @Volatile private var INSTANCE: AppDatabase? = null

        fun getInstance(context: Context): AppDatabase {
            return INSTANCE ?: synchronized(this) {
                Room.databaseBuilder(
                    context.applicationContext,
                    AppDatabase::class.java,
                    "registrasi_siswa_db"
                )
                .fallbackToDestructiveMigration()
                .build().also { INSTANCE = it }
            }
        }
    }
}

@Database(entities = [Siswa::class], version = 2) — Mendaftarkan semua entitas (tabel). version harus dinaikkan setiap kali struktur tabel berubah.

abstract class ... : RoomDatabase() — Room mengimplementasikan method abstract menggunakan kode yang dihasilkan KSP.

Singleton: @Volatile memastikan tiap thread membaca dari memori utama. synchronized(this) mencegah dua thread membuat instance bersamaan. Pola INSTANCE ?: synchronized { ... } disebut double-checked locking.

fallbackToDestructiveMigration() — Jika versi naik tanpa Migration, Room menghapus dan membuat ulang tabel daripada crash. Cocok untuk fase development.

data/repository/SiswaRepository.kt

class SiswaRepository(private val dao: SiswaDao) {
    fun getAllSiswa(): Flow<List<Siswa>> = dao.getAllSiswa()
    fun searchSiswa(query: String): Flow<List<Siswa>> = dao.searchSiswa(query)
    fun getSiswaCount(): Flow<Int> = dao.getSiswaCount()
    suspend fun insertSiswa(siswa: Siswa): Long = dao.insertSiswa(siswa)
    suspend fun updateSiswa(siswa: Siswa) = dao.updateSiswa(siswa)
    suspend fun deleteSiswa(siswa: Siswa) = dao.deleteSiswa(siswa)
    suspend fun getSiswaById(id: Int): Siswa? = dao.getSiswaById(id)
}

Saat ini Repository hanya mendelegasikan ke DAO. Nilainya terasa lebih jelas ketika ada beberapa sumber data (Room + API + cache) — semua logika penggabungan cukup ditulis sekali di sini.

viewmodel/SiswaViewModel.kt

val siswaList: StateFlow<List<Siswa>> = _searchQuery
    .flatMapLatest { query ->
        if (query.isBlank()) repository.getAllSiswa()
        else repository.searchSiswa(query)
    }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())

flatMapLatest — Setiap kali _searchQuery memancarkan nilai baru (user mengetik), subscription Flow sebelumnya dibatalkan dan diganti dengan Flow baru. Mencegah hasil pencarian lama datang setelah hasil baru.

val uiState: StateFlow<SiswaUiState> = combine(
    siswaList, _searchQuery, _isLoading, _uiMessages, totalSiswa
) { list, query, loading, messages, total ->
    SiswaUiState(
        siswaList = list, searchQuery = query, isLoading = loading,
        errorMessage = messages.first, successMessage = messages.second, totalSiswa = total
    )
}.stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), SiswaUiState())

combine(5 flows) — Menggabungkan 5 Flow berbeda menjadi satu SiswaUiState. Setiap kali salah satu berubah, combine mengeksekusi lambda untuk membentuk state baru.

SharingStarted.WhileSubscribed(5000) — Flow tetap aktif 5 detik setelah subscriber terakhir berhenti. Ini menangani rotasi layar tanpa query ulang ke database.

fun tambahSiswa(nama: String, email: String) {
    if (nama.isBlank()) {
        _uiMessages.value = Pair("Nama tidak boleh kosong", null)
        return
    }
    viewModelScope.launch {
        _isLoading.value = true
        try {
            repository.insertSiswa(Siswa(nama = nama.trim(), email = email.trim()))
            _uiMessages.value = Pair(null, "${nama.trim()} berhasil ditambahkan")
        } catch (e: Exception) {
            _uiMessages.value = Pair("Gagal: ${e.message}", null)
        } finally {
            _isLoading.value = false
        }
    }
}

Validasi sinkron dilakukan sebelum viewModelScope.launch karena tidak butuh database. try/catch/finally memastikan _isLoading selalu di-reset ke false meski terjadi error.

class Factory(private val repository: SiswaRepository) : ViewModelProvider.Factory {
    override fun <T : ViewModel> create(modelClass: Class<T>): T {
        @Suppress("UNCHECKED_CAST")
        return SiswaViewModel(repository) as T
    }
}

Factory diperlukan karena ViewModel tidak bisa memiliki constructor dengan parameter selain SavedStateHandle secara default. Factory menyuntikkan SiswaRepository saat ViewModel dibuat.

private val avatarColorPalette = listOf(
    Color(0xFFFF6B6B), Color(0xFF4ECDC4), Color(0xFF45B7D1), ...
)
private fun getAvatarColor(index: Int): Color = avatarColorPalette[index % avatarColorPalette.size]

private fun getInitials(nama: String): String {
    val words = nama.trim().split("\\s+".toRegex()).filter { it.isNotBlank() }
    return when {
        words.size >= 2 -> "${words[0].first().uppercaseChar()}${words[1].first().uppercaseChar()}"
        words.size == 1 && words[0].length >= 2 -> words[0].take(2).uppercase()
        else -> "?"
    }
}

Modulo index % palette.size memastikan indeks tidak pernah melebihi ukuran list warna. getInitials mengambil inisial dari kata pertama dan kedua, dengan fallback untuk nama satu kata atau string kosong.

MainActivity.kt

val db = AppDatabase.getInstance(this)
val repository = SiswaRepository(db.siswaDao())
val viewModel = ViewModelProvider(this, SiswaViewModel.Factory(repository))[SiswaViewModel::class.java]

setContent {
    MaterialTheme {
        Surface {
            var currentScreen by remember { mutableStateOf("list") }
            var siswaToEdit by remember { mutableStateOf<Siswa?>(null) }

            when (currentScreen) {
                "list" -> SiswaListScreen(
                    viewModel = viewModel,
                    onAddClick = { siswaToEdit = null; currentScreen = "form" },
                    onEditClick = { siswa -> siswaToEdit = siswa; currentScreen = "form" }
                )
                "form" -> FormSiswaScreen(
                    viewModel = viewModel,
                    siswaToEdit = siswaToEdit,
                    onBack = { currentScreen = "list" }
                )
            }
        }
    }
}

Manual Dependency Injection: Rantai AppDatabase → SiswaRepository → SiswaViewModel dibangun di MainActivity. ViewModelProvider(this, Factory(repository)) memastikan ViewModel bertahan melewati rotasi layar.

Navigasi berbasis state: Alih-alih Navigation Compose, proyek ini menggunakan variabel currentScreen: String + blok when. Sederhana dan cukup untuk dua layar. siswaToEdit adalah "payload" yang dikirim ke form saat mode edit — jika null berarti tambah baru.
